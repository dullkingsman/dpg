# DECLARATIVE PG — RFC

# RFC: Declarative PG (DPG)

**Status:** Draft — Proposed Specification

**Version:** 0.8.0

**Target:** PostgreSQL 14+

**File Extension:** `.dpg`

> **One-line definition:** DPG is a declarative, state-based superset of PostgreSQL SQL that compiles to idiomatic PG DDL. You describe what your database *should be*; the compiler figures out what needs to change.
> 

---

# Part I — Motivation & Philosophy

## 1.1 The Problem With Imperative Schema Management

PostgreSQL DDL is fundamentally imperative. To change a database you issue commands: `CREATE TABLE`, `ALTER TABLE ADD COLUMN`, `DROP INDEX`. This means the SQL files in your repository describe *actions taken at a point in time*, not the *current intended state* of the database. The result is a well-known set of problems:

- **Drift.** A production database that has been patched, hotfixed, and manually altered for years often no longer matches the migration history in version control.
- **No single source of truth.** To understand the current schema you must mentally replay every migration file, in order, from the beginning.
- **Idempotency is an illusion.** Running a migration file twice will fail or corrupt state.
- **Redundant context.** SQL forces you to repeat context that is already structurally known. `ALTER TABLE public.users ADD CONSTRAINT ...` re-states the schema and table name in every alteration command even though the context is known.

## 1.2 The DPG Answer

DPG flips the model. Instead of writing commands, you write a *description of the desired state*. The DPG compiler compares that description against the known schema — via a committed snapshot or a live catalog introspection — and generates the precise, minimal, safe set of PG DDL statements required to reach that state. If the database already matches the description, the output is empty. No-ops are a first-class result.

## 1.3 Why SQL Familiarity Is The Central Design Constraint

The single most important property of DPG is that it must feel like reading and writing PostgreSQL SQL. Not "inspired by SQL." Not "similar to SQL." Actual SQL, with structural scaffolding added and imperative verbs removed.

This is not an aesthetic preference. It is a practical necessity:

- PostgreSQL DBAs can audit `.dpg` files without learning a new language.
- All existing PG tooling — syntax highlighters, linters, documentation, years of institutional knowledge — remains directly applicable.
- The learning curve for a team adopting DPG is proportional only to the small set of structural additions DPG introduces, not to an entirely new DSL vocabulary.

Every design decision is evaluated against this constraint first. The default answer to "should we invent new syntax here?" is no.

## 1.4 Prior Art and Why DPG Is Different

**Atlas ([ariga.io](http://ariga.io))** uses HCL as its schema description language. HCL is clean but is a foreign DSL to a PG developer. You must learn a new vocabulary to describe objects you already know how to write in SQL. DPG's differentiator is that its object definitions *are* PostgreSQL SQL, stripped of verbs.

**Prisma Schema** invents parallel concepts for PG objects (`@id` instead of `PRIMARY KEY`, `@relation` instead of `FOREIGN KEY`). PG-specific features that Prisma doesn't model are simply inaccessible. DPG does not invent parallel concepts.

**Flyway / Liquibase / Sqitch** are migration-based, not declarative. They manage the history of changes, not the desired state.

**The DPG position:** PG SQL is already a nearly complete declarative schema language. The only things missing are structural scaffolding to remove redundancy and a diff engine to translate state changes into safe migrations. DPG adds those two things and nothing else.

## 1.5 Core Tenets

**Tenet 1 — Full PostgreSQL feature parity.** DPG must be capable of expressing anything that raw PostgreSQL DDL can express. If a PG feature cannot be declared in DPG, that is a bug in DPG, not an out-of-scope request.

**Tenet 2 — Prefer PG syntax exactly.** When PostgreSQL already has a declarative way to express something, DPG uses it verbatim. DPG removes imperative verbs and adds structural scoping, but does not invent new keywords for concepts PG already names well.

**Tenet 3 — Standard SQL / PG-extension boundary is tracked internally.** The compiler knows which constructs are ISO/IEC 9075 Standard SQL and which are PG-specific. Users never annotate portability. The compiler surfaces this via the `dpg portability` command.

**Tenet 4 — Offline-first diffing.** DPG does not require a live database connection to generate a migration. The primary workflow compares `.dpg` source files against a committed schema snapshot. Live catalog introspection is available for verification and bootstrap, but not required for day-to-day operation.

**Tenet 5 — The `{ }` block holds only what PG SQL cannot.** The native PG SQL definition of an object — its column list, its options, its clauses, its dollar-quoted body — is written exactly as PG SQL dictates. The trailing `{ }` block exists *exclusively* for things PG SQL expresses as separate DDL statements (indexes, grants, policies, comments, per-column storage attributes) and for DPG lifecycle directives (`RENAMED FROM`, `PROTECTED`, `DEPRECATED`). Nothing that has a natural place in PG SQL's own syntax is moved into the `{ }` block.

# Part II — Project Structure and Configuration

## 2.1 Overview

A DPG project maps onto the physical topology of a PostgreSQL deployment: one or more **clusters**, each containing one or more **databases**. The directory structure encodes this topology. `.dpg` source files are scoped to their containing database automatically by directory placement — no database header keyword appears in source files.

## 2.2 Directory Layout

```
myproject/
├── dpg.toml                          # Root: DPG tool configuration
│
├── production/                       # Cluster directory
│   ├── dpg.toml                      # Cluster config (connection, options)
│   │
│   ├── cluster/                      # Cluster-level objects (roles, tablespaces, FDWs)
│   │   ├── roles.dpg
│   │   └── tablespaces.dpg
│   │
│   ├── myapp/                        # Database directory
│   │   ├── dpg.toml                  # Database config (name, default_schema)
│   │   ├── extensions.dpg
│   │   └── schemas/
│   │       ├── public/
│   │       │   ├── types.dpg
│   │       │   ├── tables/
│   │       │   │   ├── users.dpg
│   │       │   │   └── orders.dpg
│   │       │   ├── views.dpg
│   │       │   └── functions.dpg
│   │       └── analytics/
│   │           └── ...
│   │
│   └── analytics_db/
│       ├── dpg.toml
│       └── ...
│
├── staging/
│   ├── dpg.toml
│   └── ...
│
└── .dpg/
    └── snapshots/
        └── production/
            ├── myapp.json
            └── analytics_db.json
```

## 2.3 The Root dpg.toml

```toml
[compiler]
default_drop_behavior = "restrict"   # restrict | cascade
concurrent_indexes    = true

[linter]
warn_on_deprecated            = true
require_column_comments       = false
forbid_hardcoded_passwords    = true
max_columns_per_table         = 50
warn_on_scalar_merge_conflict = true

[snapshots]
directory = ".dpg/snapshots"
```

## 2.4 Cluster Configuration: `<cluster>/dpg.toml`

One file per cluster, placed inside the cluster directory.

```toml
# production/dpg.toml

[cluster]
name                = "production"
cluster_objects_dir = "cluster"      # reserved; no database may share this name

# Inline PostgreSQL connection string for the primary node.
# Mutually exclusive with link.
url = "postgresql://pguser@primary.prod.internal:5432/postgres"

# Alternatively, a secrets-provider URI resolved at runtime:
# link = "env:PRIMARY_DB_URL"

[cluster.options]
snapshot_on_apply = true
```

**Connection fields:** `url` is an inline PostgreSQL connection string. `link` is a secrets-provider URI (e.g. `env:VAR_NAME`) resolved at connection time. Both `url` and `link` are mutually exclusive. Either may be omitted for offline-only use cases (`dpg plan`, `dpg diff`). Commands that connect to the database (`dpg apply`, `dpg verify`, `dpg dump`) require one to be set.

## 2.5 Database Configuration: `<cluster>/<db>/dpg.toml`

One file per database, placed inside the database directory.

```toml
# production/myapp/dpg.toml

[database]
name           = "myapp"
default_schema = "public"
```

All `.dpg` files descending from `myapp/` are automatically scoped to the `myapp` database on the `production` cluster.

## 2.6 Cluster-Level Objects

Roles, tablespaces, and custom foreign data wrappers belong to the cluster. They are declared in `.dpg` files inside the cluster objects directory (`cluster/` by default). The resolved directory name is reserved — no database within the cluster may share it.

## 2.7 Block Merge Conflict Resolution

When the same object is declared across multiple files:

- **Set-valued properties** (columns, constraints, indices, policies, triggers, grants, revocations, column blocks): union of all declarations. Identical duplicate entries are silently deduplicated. Conflicting entries (same name, different definition) are a compiler error.
- **Scalar properties** (owner, comment, tablespace, rls flags, protected, deprecated, drop behavior, renamed-from): last-declaration-wins, ordered alphabetically by full file path. Deterministic and machine-consistent. Linter warns on scalar conflicts by default.

# Part III — Language Fundamentals

## 3.1 The Two-Part Syntax Model

Every DPG object definition has at most two parts:

**Part 1 — The native PG SQL definition.** Written exactly as PostgreSQL SQL dictates: same keywords, same clause ordering, same syntax, same dollar-quoting for function bodies. For objects with a column list, this uses `( )`. For views, the `AS query` clause. For functions and procedures, the full PG SQL signature including all attributes and the `AS $$...$$` dollar-quoted body. The PG SQL clause ordering is never altered.

**Part 2 — The DPG structural block `{ }`.** An optional trailing block containing only things that PG SQL expresses as *separate DDL statements* — indexes, policies, triggers, grants, per-column storage attributes, comments — and DPG lifecycle directives (`RENAMED FROM`, `PROTECTED`, `DEPRECATED`, `DROP CASCADE`). Nothing from Part 1 is moved here. Nothing from here disrupts Part 1.

When in doubt: if PG SQL writes it as part of `CREATE OBJECT`, it is Part 1. If PG SQL writes it as a separate statement (`CREATE INDEX`, `GRANT`, `COMMENT ON`, `ALTER TABLE ... SET STATISTICS`), it is Part 2.

## 3.2 The No-Verb Mandate

`CREATE`, `ALTER`, and `DROP` are illegal in DPG source at the declaration level. The exceptions are inside dollar-quoted function bodies (`AS $$...$$`), and inside `MIGRATE REMOVE { }` blocks on enums — both are passthrough content the compiler does not interpret.

## 3.3 Structural Scoping

The `{ }` block of any container provides scope. Nested object definitions inherit their context:

```
SCHEMA analytics {
    TABLE events (
        id         BIGINT GENERATED ALWAYS AS IDENTITY,
        created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
        CONSTRAINT pk_events PRIMARY KEY (id)
    )
    {
        INDICES {
            idx_ts (created_at);    -- ON TABLE events IN SCHEMA analytics: inferred
        }
    }
}
```

Schemas have no `( )` list. Their `{ }` block directly holds all schema-level attributes and nested objects.

## 3.4 Semicolons as Statement Terminators

**Rule:** A semicolon terminates any definition not followed by a `( )` or `{ }` block. When a definition is followed by a block, the closing `)`, `$$`, `$tag$`, or `}` is its terminator and no additional semicolon precedes the following `{ }`.

**For functions and procedures:** The dollar-quoted body ends with `$$;` or `$tag$;`. This semicolon terminates the complete PG SQL function definition. An optional trailing `{ }` block follows immediately after, per the standard `; {` rule:

```
FUNCTION foo() RETURNS TEXT LANGUAGE sql STABLE
AS $$
    SELECT 'hello';
$$;
{
    COMMENT "Simple example function";
    GRANTS { EXECUTE TO app_service; }
}
```

**Without a trailing block:** The `$$;` is the complete declaration. No further punctuation:

```
FUNCTION foo() RETURNS TEXT LANGUAGE sql STABLE
AS $$
    SELECT 'hello';
$$;
```

**For tables (closing `)`):** No semicolon between `)` and `{`:

```
TABLE users (
    id BIGINT GENERATED ALWAYS AS IDENTITY,
    CONSTRAINT pk_users PRIMARY KEY (id)
) WITH (fillfactor = 90)
{
    INDICES { idx_email (email); }
}
```

**For views, enums, publications (statement ends with `;`):** Trailing `{ }` follows the semicolon:

```
VIEW active_users AS
    SELECT id, email FROM users WHERE status = 'active';
{
    GRANTS { SELECT TO app_readonly; }
}

ENUM invoice_status ('draft', 'sent', 'paid', 'void');
{
    COMMENT "Billing lifecycle states";
}
```

**Table with no sub-objects — semicolon after last PG SQL clause:**

```
TABLE simple_log (
    id      BIGINT GENERATED ALWAYS AS IDENTITY,
    message TEXT NOT NULL,
    CONSTRAINT pk_simple_log PRIMARY KEY (id)
);
```

## 3.5 Macro Preprocessing

DPG includes a source-level text preprocessor that runs before any parsing. Macros allow named, reusable fragments to be defined once and spread across column lists or block bodies.

### Defining a macro

```
MACRO common_timestamps (
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at TIMESTAMPTZ
)

MACRO audit_block {
    OWNER "app_admin";
    ENABLE ROW LEVEL SECURITY;
}
```

Two body styles:
- **Paren-body** `(...)` — used to spread column definitions into a `( )` column list.
- **Brace-body** `{...}` — used to spread directives into a `{ }` block.

`MACRO` declarations are removed from the output and generate no SQL. They must appear at the top level of a `.dpg` file (not inside blocks).

### Spreading a macro

The `...name` spread operator expands a macro inline at the point of use:

```
TABLE accounts (
    id      UUID NOT NULL DEFAULT gen_random_uuid(),
    ...common_timestamps,
    CONSTRAINT pk_accounts PRIMARY KEY (id)
)
{
    ...audit_block
}
```

The compiler expands this to the same result as if the fragment were written verbatim.

**Rules:**
- A paren-body macro may only be spread inside a `( )` list.
- A brace-body macro may only be spread inside a `{ }` block.
- Spreading an undefined macro name is a compile-time error.
- Macros are file-scoped — they are not shared across files unless redeclared. (Cross-file macro sharing is deferred.)
- `MACRO` is a DPG preprocessor keyword, not a PostgreSQL keyword. It does not violate the No-Verb Mandate (§3.2).

## 3.6 Dollar-Quote Parsing

The parser handles `$$...$$` and `$tag$...$tag$` as opaque string delimiters. When the parser encounters `AS $$`, it scans forward for the matching `$$` token. Everything between the delimiters is treated as opaque text — no brace counting, no keyword interpretation, no SQL parsing. The closing `$$` followed by `;` ends Part 1 of the definition. Named dollar-quoting (`$body$`, `$func$`, `$sql$`, any `$identifier$`) is fully supported and the parser matches the opening and closing tags exactly.

This means function bodies may freely contain any content — SQL DML, PL/pgSQL blocks, Python code, Perl, JavaScript, nested dollar-quoted strings — without any escaping.

## 3.7 Dual Definition Modes

For every object type that admits multiples, DPG supports two equivalent modes freely mixed:

**Plural block (Mode A):** Entity keyword omitted inside the scope.

```
INDICES {
    idx_email (email);
    idx_name  (last_name, first_name);
}
```

**Singular keyword (Mode B):** Singular entity keyword precedes each definition.

```
INDEX idx_email (email);
INDEX idx_name  (last_name, first_name);
```

Both are semantically identical and merged by the compiler.

## 3.8 Block Merging

Same-type blocks at the same scope level across multiple files are merged before compilation. Conflict resolution per §2.7.

## 3.9 Dependency Ordering — Topological Sort

The compiler builds a dependency graph of all declared objects before emitting any SQL. Every reference from one object to another creates a directed edge. The compiler performs a topological sort and emits DDL in dependency order.

**Circular dependencies:** Two tables with mutual FKs require at least one FK to be `DEFERRABLE`. The compiler emits both tables first, then the circular FK as a subsequent `ALTER TABLE ADD CONSTRAINT DEFERRABLE`. If a cycle exists with no `DEFERRABLE` FK, the compiler errors with a clear message identifying the cycle.

**Cross-file references:** Objects may freely reference objects declared in other files. All references are resolved across the full merged object graph before sorting.

# Part IV — Offline-First Diffing and the Introspection Engine

## 4.1 The Primary Workflow: Schema-to-Schema Diffing

```sql
[.dpg source files]  ──compile──►  [Desired State]
                                          │
                                          ▼  diff
[.dpg/snapshots/prod/myapp.json]  ──────►│
                                          ▼
                                   [Migration SQL]
```

The snapshot is committed to version control. When `dpg apply` succeeds, the snapshot is updated automatically.

**Why snapshot-first:**

- Migrations are reviewable in pull requests before any database is touched.
- CI does not require a production database connection.
- `dpg verify` checks the live catalog against the snapshot to detect drift.
- Multi-environment promotion applies the same pre-computed migration to staging and production.

## 4.2 Snapshot Format

The snapshot is human-readable JSON representing the compiler's normalised view of the database. The `grants` array in each object reflects only DPG-declared grants — it is not a complete representation of the database's privilege state. Column-level grants appear nested within each column's representation.

```json
{
  "dpg_version": "0.1.0",
  "cluster": "production",
  "database": "myapp",
  "applied_at": "2025-09-15T14:32:00Z",
  "source_revision": "a3f7c91",
  "schemas": {
    "public": {
      "tables": {
        "users": {
          "owner": "app_role",
          "comment": "Primary identity store",
          "rls_enabled": true,
          "columns": {
            "id": {
              "type": "bigint",
              "identity": { "generation": "always", "start": 1, "increment": 1 },
              "nullable": false
            },
            "email": {
              "type": "text",
              "nullable": false,
              "comment": "Verified email address",
              "statistics_target": 300,
              "grants": [
                { "grantee": "reporting_role", "privileges": ["SELECT"] }
              ]
            }
          },
          "constraints": {
            "pk_users": { "type": "primary_key", "columns": ["id"] },
            "uq_users_email": { "type": "unique", "columns": ["email"] }
          },
          "indexes": {
            "idx_users_email": {
              "method": "btree",
              "columns": [{ "name": "email", "direction": "asc" }],
              "unique": false,
              "predicate": null
            }
          },
          "grants": [
            { "grantee": "app_readonly", "privileges": ["SELECT"] },
            { "grantee": "app_service",  "privileges": ["SELECT", "INSERT", "UPDATE"] }
          ]
        }
      },
      "functions": {
        "get_user(text)": {
          "return_type": "users",
          "language": "plpgsql",
          "attributes": ["stable", "security_definer"],
          "set_options": { "search_path": "public" },
          "body_hash": "sha256:a3f7c91...",
          "comment": "Fetch a user by verified email address",
          "grants": [{ "grantee": "app_service", "privileges": ["EXECUTE"] }]
        }
      }
    }
  }
}
```

**Function snapshot note:** Function bodies are stored as a hash (`body_hash`) rather than the full text. The hash is used for change detection — if the hash changes, the compiler emits `CREATE OR REPLACE FUNCTION`. The full body text is not stored in the snapshot; it lives in the `.dpg` source files.

## 4.3 Migration Generation Modes

```bash
dpg plan                                      # .dpg source vs. committed snapshot (offline)
dpg plan --live                               # .dpg source vs. live database catalog
dpg diff --from schemas/v1/ --to schemas/v2/  # diff two DPG source directories (no snapshot)
```

**`dpg plan`** is the primary workflow. It reads committed `.dpg` source files, compiles them to the desired IR, and diffs against the committed snapshot. No database connection required unless `--live` is passed.

**`dpg diff`** compares two DPG database-scoped source directories targeting the same logical database. Output is the same SQL migration format as `dpg plan`. No snapshot or database connection required.

## 4.4 The Introspection Engine

Used for: **Bootstrap** (`dpg dump`), **Drift detection** (`dpg verify`), **Conflict resolution**.

**`dpg verify` under the additive grants model:** Reports as drift any DPG-declared grant absent from the live catalog. Does not report extra grants present in the live catalog but absent from DPG source.

Catalog tables read (PG 14+): `pg_class`, `pg_attribute` (including `attstattarget`, `attcompression`, `attstorage`), `pg_constraint`, `pg_index`, `pg_proc`, `pg_trigger`, `pg_policy`, `pg_type`, `pg_enum`, `pg_namespace`, `pg_extension`, `pg_publication`, `pg_subscription`, `pg_foreign_table`, `pg_foreign_server`, `pg_user_mapping`, `pg_foreign_data_wrapper`, `pg_statistic_ext`, `pg_event_trigger`, `pg_collation`, `pg_operator`, `pg_opclass`, `pg_opfamily`, `pg_cast`, `pg_rewrite`, `pg_partitioned_table`, `pg_inherits`, `pg_sequence`, `pg_ts_config`, `pg_ts_dict`, `pg_ts_parser`, `pg_ts_template`. Column-level grants via `information_schema.column_privileges`.

## 4.5 Portability Analysis

`dpg portability` reports all PG-specific constructs in use with SQL standard alternatives noted where available. Optional; never a compilation gate.

# Part V — Schema & Namespace Objects

## 5.1 SCHEMA

Schemas have no `( )` list. Their `{ }` block holds all attributes and nested objects.

**PG equivalent:** `CREATE SCHEMA [IF NOT EXISTS] name [AUTHORIZATION owner]`

```
SCHEMA public {
    -- objects in the public schema
}

SCHEMA analytics {
    OWNER "analytics_role";
    COMMENT "Derived tables and event aggregations";

    TABLE events ( ... ) { ... }
}
```

**Renaming** uses `RENAMED FROM` inside the `{ }` block:

```
SCHEMA reporting {
    RENAMED FROM old_reporting;
}
```

**Compiler output:** `ALTER SCHEMA old_reporting RENAME TO reporting;`

A `{ }` block containing only `RENAMED FROM` is valid.

## 5.2 EXTENSION

**PG equivalent:** `CREATE EXTENSION [IF NOT EXISTS] name [SCHEMA schema] [VERSION version] [CASCADE]`

```
EXTENSION pgcrypto;
EXTENSION postgis SCHEMA public VERSION '3.3';
EXTENSION pg_trgm CASCADE;
```

# Part VI — Type System

## 6.1 ENUM Types

ENUM types use PostgreSQL's natural parenthesised list syntax, terminated with `;`. An optional trailing `{ }` block holds comments and value-removal migration directives.

**PG equivalent:** `CREATE TYPE name AS ENUM ('val1', 'val2', ...)`

```
ENUM user_status ('active', 'suspended', 'deleted');

ENUM invoice_status ('draft', 'sent', 'paid', 'void', 'overdue');
{
    COMMENT "Billing lifecycle states for customer invoices";
}

ENUM order_status ('pending', 'confirmed', 'shipped', 'delivered');
{
    COMMENT "Order lifecycle states";
    MIGRATE REMOVE ('cancelled') {
        UPDATE orders SET status = 'delivered' WHERE status = 'cancelled';
    }
}
```

**Adding values** emits `ALTER TYPE ... ADD VALUE`.

**Removing values** — `MIGRATE REMOVE` is mandatory and contains plain DML passthrough. The compiler's procedure:

1. Create `order_status__dpg_new` with the reduced value set
2. Execute the DML inside `MIGRATE REMOVE` within a transaction
3. Verify no rows still carry removed values — abort with error listing tables and counts if any remain
4. For each column typed as `order_status`: `ALTER COLUMN c TYPE order_status__dpg_new USING c::text::order_status__dpg_new`
5. `DROP TYPE order_status`
6. `ALTER TYPE order_status__dpg_new RENAME TO order_status`
7. On failure: `DROP TYPE IF EXISTS order_status__dpg_new`

## 6.2 Composite Types

```
SCHEMA public {
    TYPE address AS (
        street      TEXT,
        city        TEXT,
        state       CHAR(2),
        postal_code TEXT,
        country     CHAR(2)
    );
}
```

## 6.3 Range Types

```
SCHEMA public {
    TYPE float8range AS RANGE (
        SUBTYPE      = float8,
        SUBTYPE_DIFF = float8mi
    );
}
```

## 6.4 Domain Types

```
SCHEMA public {
    DOMAIN positive_integer AS INTEGER {
        DEFAULT 1;
        CONSTRAINT positive_only  CHECK (VALUE > 0);
        CONSTRAINT reasonable_max CHECK (VALUE < 1000000);
    }
}
```

## 6.5 VIRTUAL TYPE

Virtual types are DPG-native annotations with no PostgreSQL equivalent. They generate no SQL whatsoever — they exist solely to carry type information for downstream consumers such as ORM generators and type-safe query builders.

```
VIRTUAL TYPE user_state AS "active" | "suspended" | "deleted";

VIRTUAL TYPE billing.payment_method AS
    { kind: "card", last4: string, brand: string }
    | { kind: "bank_ach", routing: string }
    | { kind: "wallet" };
{
    COMMENT "Payment method discriminated union for type generation";
}
```

**Rules:**
- `VIRTUAL TYPE name AS body` — the name may be schema-qualified; defaults to the `public` schema.
- The body after `AS` is arbitrary text and is stored verbatim. The compiler does not interpret it.
- No `CREATE TYPE`, `ALTER TYPE`, or `DROP TYPE` is ever emitted for a virtual type.
- Virtual types appear in the snapshot under `kind: "virtual_type"` for round-trip consistency; `dpg plan` produces no SQL for additions, modifications, or removals.
- The `{ }` block accepts only `COMMENT`.
- Intended consumers: code-generation tools that read the DPG snapshot or IR (via `pkg/dpg`).

## 6.6 Base (Shell) Types

```sql
SCHEMA public {
    TYPE mytype (
        INPUT          = mytype_in,
        OUTPUT         = mytype_out,
        INTERNALLENGTH = 16,
        ALIGNMENT      = double
    );
}
```

# Part VII — Tables

## 7.1 The Table Syntax Model

**Part 1 — The `( )` column list:** Pure PostgreSQL `CREATE TABLE` syntax. PG SQL clause modifiers appear after `)` in PG SQL's own ordering:

```
TABLE name (
    column_list,
    CONSTRAINT ...
) [INHERITS (parent)] [PARTITION BY method (col)] [WITH (options)] [TABLESPACE name]
```

**Part 2 — The `{ }` block:** Holds everything PG SQL expresses as separate DDL statements (indexes, policies, triggers, grants, per-column attributes, comments, owner), plus DPG lifecycle directives (`RENAMED FROM`, `PROTECTED`, `DEPRECATED`, `DROP CASCADE`).

### Inline vs. table-level constraints

Both forms are accepted in DPG source:

```
-- Inline (preferred for single-column constraints)
TABLE accounts (
    id   UUID    NOT NULL DEFAULT gen_random_uuid() PRIMARY KEY,
    slug TEXT    NOT NULL UNIQUE,
    org  UUID    NOT NULL REFERENCES organisations (id) ON DELETE CASCADE
);

-- Named inline
TABLE accounts (
    id UUID NOT NULL DEFAULT gen_random_uuid() CONSTRAINT pk_accounts PRIMARY KEY
);

-- Table-level (required for multi-column constraints)
TABLE order_items (
    order_id   BIGINT NOT NULL,
    product_id BIGINT NOT NULL,
    CONSTRAINT pk_order_items PRIMARY KEY (order_id, product_id)
);
```

The compiler always emits single-column `PRIMARY KEY`, `UNIQUE`, `REFERENCES`, and column-level `CHECK` constraints inline in the generated `CREATE TABLE` statement. Multi-column constraints remain table-level.

### PRIMARY KEY implies NOT NULL

PostgreSQL enforces that every `PRIMARY KEY` column is implicitly `NOT NULL`. The DPG compiler applies the same inference:

- In source, writing `NOT NULL` on a PK column is accepted but redundant.
- The compiler **does not** emit a redundant `NOT NULL` clause for PK columns in generated DDL.
- The compiler **does not** emit a spurious `ALTER COLUMN SET NOT NULL` for PK columns when diffing.

**With no sub-objects:**

```
TABLE simple_log (
    id      BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    message TEXT NOT NULL
);
```

**With sub-objects:**

```
TABLE users (
    id         BIGINT GENERATED ALWAYS AS IDENTITY,
    email      TEXT NOT NULL,
    status     user_status NOT NULL DEFAULT 'active',
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    CONSTRAINT pk_users       PRIMARY KEY (id),
    CONSTRAINT uq_users_email UNIQUE (email)
)
{
    COMMENT "Primary identity store for the application";
    OWNER   "app_role";

    COLUMN email {
        COMMENT    "Verified email address. Unique across all users.";
        STATISTICS 300;
        GRANTS     { SELECT TO reporting_role; }
    }

    COLUMN status {
        STATISTICS 500;
    }

    COLUMN created_at {
        STATISTICS 200;
    }

    INDICES {
        idx_email  (email);
        idx_status (status) WHERE (status != 'deleted');
    }

    ENABLE ROW LEVEL SECURITY;

    POLICIES {
        view_self FOR SELECT USING (id = auth.uid());
    }

    TRIGGERS {
        after_email_change AFTER UPDATE OF email
            FOR EACH ROW
            WHEN (OLD.email IS DISTINCT FROM NEW.email)
            EXECUTE FUNCTION notify_email_change();
    }

    GRANTS {
        SELECT, INSERT, UPDATE TO app_service;
        SELECT                 TO app_readonly;
    }

    REVOCATIONS {
        ALL PRIVILEGES FROM PUBLIC;
    }
}
```

**With PG SQL clause modifiers:**

```
TABLE large_events (
    id      BIGINT GENERATED ALWAYS AS IDENTITY,
    payload JSONB,
    CONSTRAINT pk_large_events PRIMARY KEY (id)

{
```

## 7.2 The COLUMN Reference Block

Inside a table's `{ }` block, `COLUMN name { }` references an existing column and holds attributes that PG SQL expresses as `ALTER TABLE ... ALTER COLUMN` statements. Per Tenet 5, these belong in the `{ }` block.

**Complete list of valid attributes inside `COLUMN name { }`:**

| Attribute | PG SQL equivalent |
| --- | --- |
| `COMMENT "text"` | `COMMENT ON COLUMN t.c IS '...'` |
| `STATISTICS n` | `ALTER TABLE t ALTER COLUMN c SET STATISTICS n` |
| `COMPRESSION method` | `ALTER TABLE t ALTER COLUMN c SET COMPRESSION m` |
| `STORAGE type` | `ALTER TABLE t ALTER COLUMN c SET STORAGE s` |
| `DEPRECATED "msg"` | `COMMENT ON COLUMN t.c IS '[DEPRECATED] msg'` |
| `USING expr` | `ALTER TABLE t ALTER COLUMN c TYPE ... USING expr` |
| `RENAMED FROM old_name` | `ALTER TABLE t RENAME COLUMN old_name TO new_name` |
| `GRANTS { }` | `GRANT privilege (col) ON TABLE t TO role` |
| `REVOCATIONS { }` | `REVOKE privilege (col) ON TABLE t FROM role` |

**Validation rules:**

- Every `COLUMN name { }` must reference a column that exists in the `( )` list. Compiler errors if the name does not match.
- After a rename, the `COLUMN` block uses the new name; `RENAMED FROM` carries the old name.
- After any column rename, all index and constraint declarations must use the new column name. The compiler validates and errors on mismatches.
- `COLUMN` blocks do not declare new columns — new columns are declared only in the `( )` list.
- Plural form `COLUMNS { name1 { ... } name2 { ... } }` is also valid.

## 7.3 Column-Level Grants

Column-level grants use the same `GRANTS { }` syntax inside the `COLUMN name { }` block. The column scope is inferred — the compiler emits `GRANT privilege (col) ON TABLE t TO role`.

```
TABLE users ( id BIGINT ..., email TEXT, ssn TEXT, ... )
{
    COLUMN email {
        GRANTS {
            SELECT TO reporting_role;
            SELECT TO analytics_role;
        }
    }

    COLUMN ssn {
        STORAGE extended;
        GRANTS     { SELECT TO compliance_role; }
        REVOCATIONS { ALL PRIVILEGES FROM PUBLIC; }
    }

    GRANTS {
        SELECT, INSERT, UPDATE TO app_service;   -- table-level grant
    }
}
```

Column-level grants follow the same additive model as table-level grants: DPG only emits `GRANT`; it never auto-revokes.

## 7.4 Column Renaming

Column renames are DPG lifecycle directives and live in the `{ }` block inside the `COLUMN name { }` block, using `RENAMED FROM old_name`. The new name appears in both the `( )` list and the `COLUMN` block. The old name appears only inside `RENAMED FROM`.

```
TABLE users (
    email_address TEXT NOT NULL,   -- new name in ( ) list
    ...
    CONSTRAINT uq_users_email UNIQUE (email_address)
)
{
    COLUMN email_address {
        RENAMED FROM email;
        COMMENT "Verified email address";
    }
}
```

**Compiler resolution algorithm:**

1. Compiler sees `email_address` in the `( )` list and `COLUMN email_address { RENAMED FROM email; }` in the `{ }` block.
2. Looks up `email` in the snapshot. If `email` exists and `email_address` does not, this is a rename.
3. Emits `ALTER TABLE users RENAME COLUMN email TO email_address` — not `ADD COLUMN email_address`.
4. Validates that all index and constraint declarations in `{ }` reference `email_address`, not `email`. Errors on mismatch.
5. If `email` does not exist in the snapshot, `RENAMED FROM` without a snapshot match is a compiler error — remove `RENAMED FROM` for genuinely new columns.

**`RENAMED FROM` applies to all renameable objects:**

- Table: `RENAMED FROM old_table_name;` inside table `{ }` block
- Schema: `RENAMED FROM old_schema_name;` inside schema `{ }` block
- Column: `RENAMED FROM old_col_name;` inside `COLUMN new_name { }` inside table `{ }` block

## 7.5 The CONSTRAINT Block in `{ }`

`CONSTRAINT` in the `{ }` block handles constraints that cannot appear in `CREATE TABLE` (specifically `NOT VALID` constraints) and provides alternative placement for any constraint.

Constraints are identified by name across both `( )` and `{ }` — merged into a single logical set. Same name + same definition = deduplicated. Same name + conflicting definition = compiler error.

```
TABLE orders (
    id     BIGINT GENERATED ALWAYS AS IDENTITY,
    amount NUMERIC(10,2) NOT NULL,
    CONSTRAINT pk_orders PRIMARY KEY (id)
)
{
    CONSTRAINT ck_amount_positive CHECK (amount > 0) NOT VALID;

    CONSTRAINT fk_account FOREIGN KEY (account_id)
        REFERENCES accounts (id)
        ON DELETE CASCADE;
}
```

**`NOT VALID` lifecycle:** Compiler emits `ALTER TABLE ADD CONSTRAINT ... NOT VALID`. When `NOT VALID` is removed in a later migration, compiler emits `ALTER TABLE VALIDATE CONSTRAINT name`. A validated constraint may be moved to the `( )` list — the compiler identifies it by name and treats it as already existing.

## 7.8 Partitioned Tables

```
SCHEMA public {
    TABLE events (
        id         BIGINT GENERATED ALWAYS AS IDENTITY,
        tenant_id  UUID NOT NULL,
        event_type TEXT NOT NULL,
        payload    JSONB,
        created_at TIMESTAMPTZ NOT NULL DEFAULT now()
    ) PARTITION BY RANGE (created_at)
    {
        INDICES {
            idx_events_tenant (tenant_id);
            idx_events_type   (event_type);
        }

        PARTITIONS {
            events_2024_q1 FOR VALUES FROM ('2024-01-01') TO ('2024-04-01');
            events_2024_q2 FOR VALUES FROM ('2024-04-01') TO ('2024-07-01');
            events_2024_q3 FOR VALUES FROM ('2024-07-01') TO ('2024-10-01');
            events_2024_q4 FOR VALUES FROM ('2024-10-01') TO ('2025-01-01');
            events_default DEFAULT;
        }
    }

    TABLE orders_by_region (
        id     BIGINT GENERATED ALWAYS AS IDENTITY,
        region TEXT NOT NULL
    ) PARTITION BY LIST (region)
    {
        PARTITIONS {
            orders_north FOR VALUES IN ('NYC', 'BOS', 'PHI');
            orders_south FOR VALUES IN ('MIA', 'ATL', 'DAL');
        }
    }
}
```

**Sub-partitioning:**

```
TABLE events ( ... ) PARTITION BY RANGE (created_at)
{
    PARTITIONS {
        events_2024 FOR VALUES FROM ('2024-01-01') TO ('2025-01-01')
            PARTITION BY LIST (region) {
                PARTITIONS {
                    events_2024_us FOR VALUES IN ('us-east', 'us-west');
                    events_2024_eu FOR VALUES IN ('eu-west', 'eu-central');
                }
            };
    }
}
```

**Partition strategy changes** require `dpg apply --approve-partition-rebuild`. The skeleton: create new table with target strategy → create all declared partitions → insert all data → verify row counts → drop original → rename → recreate all indexes, constraints, grants, and policies.

**Adding partitions** is always safe. **Removing partitions** is `DESTRUCTIVE`.

# Part VIII — Column Statistics Targets

## 8.1 What Column Statistics Are

When PostgreSQL executes `ANALYZE`, it samples each column and computes: most common values and their frequencies, a value distribution histogram, NULL fraction, and distinct value count. The query planner uses these to estimate row counts for conditions, determining join strategy and scan type. Poor statistics produce bad plans.

## 8.2 What the Statistics Target Controls

| Value | Meaning |
| --- | --- |
| `-1` | Reset to table default (normally 100 at cluster level) |
| `0` | Disable statistics collection for this column |
| `1–10000` | Explicit target; above 100 = more detail, more ANALYZE cost |

Raise the target on columns with skewed distributions, columns driving range queries, or columns where `EXPLAIN ANALYZE` shows large row-count estimate gaps. Values above 500 rarely help.

## 8.3 DPG Syntax

```
TABLE orders ( amount NUMERIC(10,2), status order_status, created_at TIMESTAMPTZ, ... )
{
    COLUMN status     { STATISTICS 500; }
    COLUMN created_at { STATISTICS 200; }
    COLUMN notes      { STATISTICS 0;   }
}
```

**Compiler output:**

```sql
ALTER TABLE orders ALTER COLUMN status     SET STATISTICS 500;
ALTER TABLE orders ALTER COLUMN created_at SET STATISTICS 200;
ALTER TABLE orders ALTER COLUMN notes      SET STATISTICS 0;
```

# Part XII — Views

## 12.1 Regular Views

The `AS query` clause is Part 1, terminated with `;`. PG SQL options appear on the signature line. An optional trailing `{ }` block holds grants, comments, and other sub-objects.

```
SCHEMA public {
    VIEW active_users AS
        SELECT id, email, created_at
        FROM users
        WHERE status = 'active' AND deleted_at IS NULL;

    VIEW user_summary (user_id, email, order_count) AS
        SELECT u.id, u.email, COUNT(o.id)
        FROM users u
        LEFT JOIN orders o ON o.user_id = u.id
        GROUP BY u.id, u.email;

    VIEW secure_user_view WITH (security_barrier = true) AS
        SELECT id, email FROM users WHERE tenant_id = current_tenant();

    VIEW active_orders AS
        SELECT * FROM orders WHERE status != 'cancelled'
        WITH LOCAL CHECK OPTION;

    RECURSIVE VIEW org_tree (id, parent_id, depth, path) AS
        SELECT id, parent_id, 0, ARRAY[id]
        FROM departments WHERE parent_id IS NULL
        UNION ALL
        SELECT d.id, d.parent_id, t.depth + 1, t.path || d.id
        FROM departments d JOIN org_tree t ON d.parent_id = t.id;

    VIEW admin_summary AS
        SELECT id, email, created_at FROM users WHERE role = 'admin';
    {
        COMMENT "Admin user summary";
        GRANTS { SELECT TO app_readonly; }
    }
}
```

**View diffing:** Query changes with identical output column list → `CREATE OR REPLACE VIEW`. Output column list changes in any way → `DROP VIEW CASCADE` then `CREATE VIEW`, classified as `DESTRUCTIVE`.

## 12.2 Materialized Views

```
SCHEMA analytics {
    MATERIALIZED VIEW daily_revenue AS

        WHERE status = 'completed'
        GROUP BY 1;

    MATERIALIZED VIEW product_stats
    WITH (fillfactor = 90)
    TABLESPACE analytics_space AS
        SELECT product_id, COUNT(*) AS purchases, AVG(price) AS avg_price
        FROM order_items
        GROUP BY product_id
    WITH NO DATA;
    {
        INDICES {
            idx_product_stats_id (product_id);
        }
        GRANTS {
            SELECT TO app_readonly;
        }
    }
}
```

# Part XIII — Functions & Procedures

## 13.1 Function Syntax Model

Functions are written in complete, unmodified PostgreSQL SQL syntax — the same syntax a developer would write in a `CREATE FUNCTION` statement, with only the `CREATE OR REPLACE` verb removed. The dollar-quoted body using `AS $$...$$` or `AS $tag$...$tag$` is Part 1. The trailing `{ }` block (Part 2) holds grants, comment, deprecated, and any other externally-attachable concerns.

```
FUNCTION name(args) RETURNS type
LANGUAGE lang
[attributes in PG SQL order]
AS $$
    function body — opaque passthrough, any content
$$;
{
    COMMENT "Description";
    GRANTS { EXECUTE TO role_name; }
    DEPRECATED "Use new_function instead";
}
```

The `$$;` ends the complete function definition (Part 1). The `{ }` block follows immediately if sub-objects are needed. If no sub-objects are needed, `$$;` alone is the complete declaration.

The compiler parses `AS $$...$$` by scanning for the matching closing delimiter. Everything between the delimiters is opaque text — no interpretation, no keyword scanning. Named dollar-quoting (`$body$`, `$func$`, `$sql$`, any `$identifier$`) is fully supported.

## 13.2 Function Examples

```
SCHEMA public {

    -- Simple SQL function, no sub-objects
    FUNCTION active_user_count() RETURNS BIGINT
    LANGUAGE sql STABLE PARALLEL SAFE
    AS $$
        SELECT COUNT(*) FROM users WHERE status = 'active';
    $$;

    -- PL/pgSQL function with grants and comment
    FUNCTION get_user(p_email TEXT) RETURNS users
    LANGUAGE plpgsql STABLE SECURITY DEFINER SET search_path = public
    AS $$
    DECLARE
        v_user users;
    BEGIN
        SELECT * INTO STRICT v_user FROM users WHERE email = p_email;
        RETURN v_user;
    EXCEPTION
        WHEN NO_DATA_FOUND THEN
            RAISE EXCEPTION 'User not found: %', p_email;
    END;
    $$;
    {
        COMMENT "Fetch a user record by verified email address";
        GRANTS { EXECUTE TO app_service; }
    }

    -- Set-returning function
    FUNCTION users_in_org(p_org_id UUID) RETURNS SETOF users
    LANGUAGE sql STABLE
    AS $$
        SELECT * FROM users WHERE org_id = p_org_id ORDER BY email;
    $$;

    -- Returns TABLE inline
    FUNCTION monthly_summary(p_year INT)
    RETURNS TABLE (month INT, revenue NUMERIC, count BIGINT)
    LANGUAGE sql STABLE
    AS $$
        SELECT
            EXTRACT(MONTH FROM created_at)::INT,
            SUM(total),
            COUNT(*)
        FROM orders
        WHERE EXTRACT(YEAR FROM created_at) = p_year
        GROUP BY 1 ORDER BY 1;
    $$;

    -- Variadic arguments
    FUNCTION log_event(p_type TEXT, VARIADIC p_tags TEXT[]) RETURNS VOID
    LANGUAGE plpgsql
    AS $$
    BEGIN
        INSERT INTO event_log (event_type, tags) VALUES (p_type, p_tags);
    END;
    $$;

    -- With default parameters
    FUNCTION paginate_users(p_limit INT DEFAULT 20, p_offset INT DEFAULT 0)
    RETURNS SETOF users
    LANGUAGE sql STABLE
    AS $$
        SELECT * FROM users LIMIT p_limit OFFSET p_offset;
    $$;

    -- Trigger function
    FUNCTION notify_email_change() RETURNS TRIGGER
    LANGUAGE plpgsql SECURITY DEFINER SET search_path = public
    AS $$
    BEGIN
        PERFORM pg_notify('email_changed', NEW.id::TEXT);
        RETURN NEW;
    END;
    $$;

    -- Named dollar-quoting (useful when body contains $$ literals)
    FUNCTION format_price(p_amount NUMERIC) RETURNS TEXT
    LANGUAGE plpgsql IMMUTABLE STRICT
    AS $func$
    BEGIN
        RETURN '$' || TO_CHAR(p_amount, 'FM999,999,990.00');
    END;
    $func$;
    {
        COMMENT "Format a numeric amount as a dollar price string";
        GRANTS { EXECUTE TO app_readonly, app_service; }
    }

}
```

## 13.3 Function Attributes Reference

All attributes appear on the signature line in PG SQL's own order, exactly as `CREATE FUNCTION` writes them.

| Attribute | Meaning |
| --- | --- |
| `VOLATILE` | Default. May modify DB; result may differ per call |
| `STABLE` | Constant within a transaction for given inputs |
| `IMMUTABLE` | Constant forever for given inputs; index-eligible |
| `STRICT` | Returns NULL if any argument is NULL |
| `SECURITY DEFINER` | Executes with owner's privileges |
| `SECURITY INVOKER` | Default. Executes with caller's privileges |
| `PARALLEL SAFE` | Safe for parallel workers |
| `PARALLEL RESTRICTED` | Parallel, leader process only |
| `PARALLEL UNSAFE` | Default. Cannot run in parallel |
| `COST n` | Estimated execution cost in cpu_operator_cost units |
| `ROWS n` | Estimated rows returned (set-returning functions) |
| `SUPPORT func` | Planner support function (PG 12+) |
| `SET param = value` | Set a GUC for the duration of the call |
| `WINDOW` | Marks as a window function |

## 13.4 Procedures

Procedures follow the same model. The procedure body uses `AS $$...$$`. Procedures may `COMMIT` mid-execution.

```
SCHEMA public {
    PROCEDURE process_settlements()
    LANGUAGE plpgsql SECURITY DEFINER
    AS $$
    DECLARE
        v_id settlements.id%TYPE;
    BEGIN
        FOR v_id IN SELECT id FROM settlements WHERE processed = false LOOP
            PERFORM settle_order(v_id);
            COMMIT;
        END LOOP;
    END;
    $$;
    {
        GRANTS { EXECUTE TO scheduler_role; }
    }
}
```

## 13.5 Aggregate Functions

Aggregates use two `( )` groups per PG SQL's `CREATE AGGREGATE` syntax — both are Part 1 per Tenet 5. The trailing `{ }` block holds grants, comments, and other externally-attachable concerns.

```
SCHEMA public {
    AGGREGATE product (DOUBLE PRECISION) (
        SFUNC    = float8mul,
        STYPE    = DOUBLE PRECISION,
        INITCOND = '1'
    )
    {
        COMMENT "Multiplicative aggregate over DOUBLE PRECISION values";
        GRANTS { EXECUTE TO app_service; }
    }

    AGGREGATE percentile_disc (DOUBLE PRECISION ORDER BY anyelement) (
        SFUNC     = ordered_set_transition,
        STYPE     = internal,
        FINALFUNC = percentile_disc_final,
        FINALFUNC_EXTRA
    )
    {
        COMMENT "Ordered-set percentile aggregate";
    }
}
```

**Aggregate diffing:** Identity is `(schema, name, input_types)`. Changes to `SFUNC`, `STYPE`, `INITCOND`, `FINALFUNC`, `COMBINEFUNC`, or `SERIALFUNC` require DROP + recreate (`DESTRUCTIVE`).

## 13.6 Function Body Diffing

The compiler stores a hash of the normalised function body in the snapshot (§4.2). Any change to the body text — even whitespace — changes the hash and causes the compiler to emit `CREATE OR REPLACE FUNCTION`. No semantic diff of procedural code is performed. This is a known accepted limitation.

# Part XV — Access Control

## 15.1 Roles

Cluster-level objects; defined in the cluster objects directory.

```
-- production/cluster/roles.dpg

ROLE app_readonly {
    NOLOGIN;
    COMMENT "Read-only access for reporting tools";
}

ROLE app_service {
    LOGIN;
    PASSWORD 'env:APP_SERVICE_PW';
    CONNECTION LIMIT 20;
    VALID UNTIL '2030-01-01';
}

ROLE app_admin {
    LOGIN;
    SUPERUSER  false;
    CREATEDB   false;
    CREATEROLE false;
    INHERIT;
    IN ROLE pg_read_all_stats;
}
```

Hardcoded passwords are rejected by the linter. Passwords must use `env:VAR_NAME` syntax.

## 15.2 GRANTS — Additive Model

DPG follows the same additive model as raw PostgreSQL: declaring a grant emits a `GRANT` statement. DPG never auto-revokes. Removing a `GRANT` entry from source emits nothing — add an explicit `REVOCATIONS { }` entry if revocation is intended.

```
TABLE orders ( ... )
{
    GRANTS {
        SELECT                 TO app_readonly;
        SELECT, INSERT, UPDATE TO app_service;
        ALL PRIVILEGES         TO app_admin;
    }

    REVOCATIONS {
        ALL PRIVILEGES FROM PUBLIC;
    }
}

SCHEMA analytics {
    GRANTS {
        USAGE TO app_readonly;
        ALL   TO analytics_admin;
    }
}

FUNCTION calculate_total(p_id UUID) RETURNS NUMERIC
LANGUAGE plpgsql STABLE
AS $$
BEGIN
    RETURN (SELECT SUM(amount) FROM line_items WHERE order_id = p_id);
END;
$$;
{
    GRANTS { EXECUTE TO app_service; }
}
```

**Default privileges:**

```
SCHEMA public {
    DEFAULT PRIVILEGES FOR ROLE app_admin {
        GRANTS {
            SELECT   ON TABLES    TO app_readonly;
            EXECUTE  ON FUNCTIONS TO app_service;
            USAGE    ON SEQUENCES TO app_service;
        }
    }
}
```

# Part XVI — Full-Text Search Objects

Mapping configuration uses `MAPPING FOR` (declarative). The compiler emits `ALTER TEXT SEARCH CONFIGURATION ... ALTER MAPPING` DDL.

```
SCHEMA public {
    TEXT SEARCH CONFIGURATION english_unaccented (COPY = pg_catalog.english) {
        MAPPING FOR hword, hword_part, word
```

# Part XX — Compiler Behavior & Migration Output

## 20.1 CLI Commands

| Command | Description |
| --- | --- |
| `dpg plan` | Diff desired state vs. snapshot. Prints SQL. No DB connection required unless `--live` is passed. |
| `dpg apply` | Run linter + diff, prompt for approval, execute SQL, update snapshot. |
| `dpg verify` | Introspect live DB and report drift vs. snapshot. Reports missing managed grants; ignores extra unmanaged grants. |
| `dpg dump` | Introspect live DB and produce initial `.dpg` source + snapshot. |
| `dpg diff --from <dir> --to <dir>` | Diff two DPG database-scoped source directories. Output is SQL migration format. No snapshot or DB required. |
| `dpg validate` | Compile and lint `.dpg` source files offline. No snapshot or DB required. `--format json` for machine-readable output. |
| `dpg portability` | Report all PG-specific constructs with standard SQL alternatives. |
| `dpg init` | Scaffold a new project with standard directory layout and `dpg.toml` files. |
| `dpg fmt` | Reformat `.dpg` source files in place. `--check` for CI gate; `--diff` for preview. |
| `dpg completion` | Generate shell completion scripts (Bash, Zsh, Fish, PowerShell). |

## 20.2 Migration Output Format

```sql
-- DPG Migration
-- Generated:       2025-09-15T14:32:00Z
-- Source revision: a3f7c91
-- Cluster:         production
-- Database:        myapp

BEGIN;

-- [source: schemas/public/tables/users.dpg:4]
CREATE TABLE public.users (
    id        BIGINT GENERATED ALWAYS AS IDENTITY CONSTRAINT "pk_users" PRIMARY KEY,
    email     TEXT NOT NULL CONSTRAINT "uq_users_email" UNIQUE
);

COMMENT ON TABLE  public.users       IS 'Primary identity store';
COMMENT ON COLUMN public.users.email IS 'Verified email address';

ALTER TABLE public.users ALTER COLUMN email SET STATISTICS 300;

GRANT SELECT         ON TABLE  public.users TO app_readonly;
GRANT SELECT (email) ON TABLE  public.users TO reporting_role;

-- [source: schemas/public/functions.dpg:12]
CREATE OR REPLACE FUNCTION public.get_user(p_email TEXT) RETURNS users
LANGUAGE plpgsql STABLE SECURITY DEFINER SET search_path = public
AS $$
DECLARE
    v_user users;
BEGIN
    SELECT * INTO STRICT v_user FROM users WHERE email = p_email;
    RETURN v_user;
END;
$$;

COMMENT ON FUNCTION public.get_user(TEXT) IS 'Fetch a user record by verified email address';
GRANT EXECUTE ON FUNCTION public.get_user(TEXT) TO app_service;

COMMIT;

-- Non-transactional steps (executed after COMMIT):
-- [source: schemas/public/tables/users.dpg:22]
CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_users_email ON public.users (email);
```

## 20.3 Idempotency Guarantee

Running `dpg apply` on a database that already matches the declared state produces zero SQL statements. Any violation is a compiler bug.

## 20.4 Safety Classification

| Class | Description | Default Behavior |
| --- | --- | --- |
| `SAFE` | No data loss possible | Applied automatically |
| `CAUTION` | Locks acquired; performance impact possible | Applied with warning logged |
| `DESTRUCTIVE` | Data loss possible | Blocked unless `--allow-destructive` or explicit acknowledgment in source |
| `MANUAL` | Cannot run inside a transaction (concurrent index creation), or describes a manual operator step (partition strategy change) | Executable `MANUAL` ops are emitted after `COMMIT` as non-transactional steps. Instruction-only `MANUAL` ops (comment-only, prefixed with `--`) are displayed in the plan but never executed — the operator must perform them manually. `--approve-partition-rebuild` is required to acknowledge them. |

```
SCHEMA public {
    TABLE invoices (
        id         UUID           NOT NULL DEFAULT gen_random_uuid(),
        account_id UUID           NOT NULL,
        status     invoice_status NOT NULL DEFAULT 'draft',
        total      positive_money NOT NULL DEFAULT 0,
        issued_at  TIMESTAMPTZ,
        due_at     TIMESTAMPTZ,
        created_at TIMESTAMPTZ    NOT NULL DEFAULT now(),
        CONSTRAINT pk_invoices         PRIMARY KEY (id),
        CONSTRAINT fk_account          FOREIGN KEY (account_id)
                                           REFERENCES accounts (id)
                                           ON DELETE CASCADE,
        CONSTRAINT ck_due_after_issued CHECK (due_at IS NULL OR due_at > issued_at)
    )
    {
        COLUMN created_at { STATISTICS 200; }

        INDICES {
            idx_invoices_account (account_id);
            idx_invoices_status  (status) WHERE (status NOT IN ('paid', 'void'));
            idx_invoices_due     (due_at) WHERE (status = 'sent');
        }

        ENABLE ROW LEVEL SECURITY;

        POLICIES {
            isolate_tenants FOR ALL
                USING (account_id = current_setting('app.account_id')::UUID);
        }

        TRIGGERS {
            after_status_change AFTER UPDATE OF status
                FOR EACH ROW
                WHEN (OLD.status IS DISTINCT FROM NEW.status)
                EXECUTE FUNCTION notify_invoice_status_change();
        }

        GRANTS {
            SELECT, INSERT, UPDATE TO app_service;
        }
    }
}
```

```
-- production/myapp/schemas/public/functions.dpg

SCHEMA public {

    FUNCTION notify_invoice_status_change() RETURNS TRIGGER
    LANGUAGE plpgsql SECURITY DEFINER
    AS $$
    BEGIN
        PERFORM pg_notify(
            'invoice_status_changed',
            json_build_object(
                'invoice_id', NEW.id,
                'old_status', OLD.status,
                'new_status', NEW.status
            )::TEXT
        );
        RETURN NEW;
    END;
    $$;

    MATERIALIZED VIEW account_billing_summary AS
        SELECT
            a.id,
            a.name,
            COUNT(i.id)                                                    AS total_invoices,
            COALESCE(SUM(i.total) FILTER (WHERE i.status = 'paid'),    0)  AS paid_total,
            COALESCE(SUM(i.total) FILTER (WHERE i.status = 'overdue'), 0)  AS overdue_total
        FROM accounts a
        LEFT JOIN invoices i ON i.account_id = a.id
        WHERE a.deleted_at IS NULL
        GROUP BY a.id, a.name
    WITH NO DATA;

}
```

# Part XXII — Deferred Features

The following are formally out of scope for the current specification and deferred to future versioned RFCs.

**Inline data seeding (`SEED { }` blocks):** Deferred to a future `dpg-seed` extension RFC. Blurs the boundary between schema management and data management.

**Minimum PostgreSQL version targeting:** Planned for v1.1. Straightforward once the compiler's internal portability annotations are in place.

**Cross-file macro sharing:** Currently, `MACRO` definitions are file-scoped — a macro defined in one `.dpg` file is not visible in other files. A future iteration will add a global macro registry so shared column sets can live in a dedicated `macros.dpg` file and be imported by any other file in the project.
